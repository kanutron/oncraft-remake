# OnCraft Remake — Patterns Index

Patterns discovered during implementation sessions.

## Patterns

| Pattern | Location | Summary |
|---|---|---|
| Chat event registry | `packages/frontend/app/composables/chat/event-registry.ts` | Single source of truth mapping SDK event types to render components, relationships, correlation keys, and default render modes. Guarded by a build-time drift test against `SDKMessage`. |
| Bridge emit payload shape | `packages/backend/src/bridge/session-bridge.ts` + `services/process-manager.ts` | Bridge must NOT emit a top-level `sessionId` field. The process-manager spreads the bridge event over its own `{ sessionId: <internal UUID>, ...event }`, and a bridge-side `sessionId` would override the internal one — routing events to a phantom session on the frontend. Only emit event-specific fields; let the PM add `sessionId`. |
| Subagent embedding (Agent tool_use) | `packages/frontend/app/composables/chat/use-subagent-correlation.ts` + `components/chat/ChatBlockToolUse.vue` + `ChatSubagentTranscript.vue` | Subagent transcripts render inline inside their parent Agent `tool_use` card. Correlation uses `(agentType, description)` from `.meta.json` matched against parent's `input.{subagent_type, description}`, with positional pairing for duplicate pairs. Live subagent turns (SDK events carrying `parent_tool_use_id`) are indexed in the store by that id and merged with hydrated (`.jsonl`) messages, deduped by `uuid`. Nested rendering reuses the reducer + resolver pipeline so nested chats match top-level behavior. |
| Bridge `tool_confirmation` payload + lifecycle | `packages/backend/src/bridge/session-bridge.ts` + `packages/frontend/app/components/chat/ChatToolConfirmation.vue` + `composables/chat/use-chat-reducer.ts` | The bridge forwards SDK `canUseTool` args verbatim and emits `{ type: 'tool_confirmation', toolUseID, toolName, toolInput, agentID?, decisionReason? }`. Frontend consumers (component props, reducer fixtures, tests) must use these exact field names — do NOT rename to `request_id`/`tool`/`input`. The component's click handler guards on `toolUseID`; a field-name drift silently disables Allow/Deny because the guard short-circuits before calling `sessionStore.reply`. Component tests MUST drive the click path and assert the POST body, not just render — a render-only test will pass under a broken guard. Lifecycle: the event is live-only (never persisted in `.jsonl` history). Event ordering: the assistant's `tool_use` block is emitted BEFORE `canUseTool` is invoked, so filtering the banner on `tool_use` presence hides it every time and the UI appears to hang. The reducer must suppress it on `tool_result` presence for the same id (i.e. the tool has actually run) — by then the tool_use card already carries the outcome. The component additionally hides locally on click (`decided` ref) so the dismissal is instant, before the SDK round-trips the tool_use event. Reply resolution must return a full SDK `PermissionResult` — allow → `{ behavior: 'allow', updatedInput }` (echo the original `toolInput`, so the bridge caches it on the pending approval), deny → `{ behavior: 'deny', message: string }`. Resolving with just `{ behavior }` fails the SDK's Zod validation ("expected string, received undefined" on `message`). |
| Rich markdown render pipeline | `packages/frontend/app/composables/chat/markdown/*` + `components/chat/RichMarkdown.vue` | Single synchronous pipeline: `markdown-it (html+linkify+GFM) → @shikijs/markdown-it (dual theme) → DOMPurify (explicit allowlist) → formatStackTraces → linkifyFilePaths` (DOMParser-guarded). Tool inputs/outputs pass through pure formatters (`formatToolInput`/`formatToolOutput`) that produce markdown strings routed through the same pipeline. Shiki init is async; `renderMarkdown` falls back to an escaped `<pre>` and the `useMarkdown` composable bumps a `version` ref once ready to trigger re-render. Composable tests that import DOMPurify need `// @vitest-environment happy-dom` as the first line because the `node` vitest project has no DOM. `RichMarkdown.vue` walks its rendered HTML via `DOMParser` to swap `.mermaid-src` placeholders for `<MermaidBlock>` and long `<pre>` blocks (> 20 lines) for `<CodeCollapsible>`. |
